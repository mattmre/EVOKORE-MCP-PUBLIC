Top-tier feature spotlight

ARCH-AEP — Align, Execute, Prove

AEP is a structured engineering methodology for agentic workflows. It organizes work into three repeating phases — Align (scope lock), Execute (severity-tiered remediation), and Prove (auditable verification evidence) — and refuses to advance a tier until the previous one closes with proof.

The triangle

Three phases. Each one feeds the next. The loop closes when evidence proves the work, not when the work looks done.

Each pass through the triangle ends with a verification log, not with a green merge button. The next pass starts only after the last tier closes.

Phase 1 — Align

The Align phase locks the scope before any code is touched. It exists because agentic workflows otherwise drift mid-execution: scope creeps, findings get re-prioritized, and the team loses its baseline.

Step 1 — Scope lock

Define the PR range. Define the time window. Freeze the refinement report or framework input. After this step, no input gets edited until Execute begins.

Step 2 — Discovery + normalization

Ingest the review framework. Normalize findings into a single backlog with unique IDs (AEP-YYYYMMDD-PRNNN-SEQ). De-duplicate, merge overlaps, assign provisional severity.

Step 3 — Parallel re-validation

Specialist agents re-validate each finding against current main. Each agent attaches exact file paths, line ranges, smallest safe fix, and S/M/L sizing.

Step 4 — Architecture synthesis

The orchestrator merges validated findings into a master backlog. Re-score using severity → impact × 1/effort → dependency order. Identify chains and blockers.

Phase 1 W3 — Context and decisions discipline

Bounded contexts get a docs/contexts/<slug>/CONTEXT.md file. Architecture decisions get an ADR in docs/adr/. ADRs only reach Status: Accepted after two Panel-of-Experts review passes. Reopening an Accepted ADR requires a new superseding ADR — the decision log is immutable history.

Phase 2 — Execute

Severity-tiered remediation. The tier order is Critical → High → Medium → Low. No tier advances until the current tier closes with verification evidence on every item.

A blocked Critical item never lets a Medium item ship. The unblock plan is itself a finding to be remediated.

Tier-close checklist (auditability)

  • All findings in the tier show verification evidence in both tracker and cycle verification log.
  • Phase summaries exist for every PR in the tier.
  • Risk memos for Critical/High items are archived.
  • No item is left in "in progress" state — items are either closed or formally deferred.

Phase 3 — Prove

Every merged fix requires auditable evidence. Verification is not a checkbox — it is an artifact.

verification-log.md

What gets logged

  • Exact command with flags
  • Exit code
  • Summary of stdout/stderr; full output on failure
  • Environment (OS, runtime version, dependencies)
  • Baseline state (git SHA, branch)
  • Any flaky tests observed
evaluation-harness.md

How orchestration quality is judged

  • Golden tasks for the agent type under test
  • Pass/fail against golden cases
  • Metrics across cycles (work density, clarification loops)
  • Persona effectiveness across sessions (via the meta-improvement cycle)

The test matrix

The test matrix is the rule for which tests to run on a given change. Baseline tests run on every change; conditional tests run when the change touches a flagged area.

Baseline (always)

  • Unit tests — full suite
  • Lint/format checks

Conditional

Change typeAdditional tests
Parser changesParser-specific tests + sample fixtures
IO / file handlingIntegration tests for read/write paths
Performance changesBenchmark or perf regression check
Security changesRelevant security tests or static analysis
Documentation-onlyLint docs links and build docs if available

Agent-specific selection

Agent typeMinimum testsExtended tests
ImplementerUnit tests for changed modulesFull suite if touching shared code
DebuggerFailing test + related testsRegression suite for the affected area
RefactorerFull unit suiteIntegration tests if API surface changed
Tester(creates tests)Run all new + existing tests
ReviewerRead-only (no execution)Optional spot-check of critical paths

Partial runs — and how to do them honestly

Sometimes the full test suite cannot run — a missing dependency, an unavailable external service, a documentation-only change. Partial runs are allowed but have a protocol:

  1. Document the reason. Explain why the full suite was skipped.
  2. Select the minimum set. Smoke tests, or tests for the affected module only.
  3. Record the selection rationale. Link to the test matrix and the specific justification.
  4. Flag for follow-up. Note in the session log if a full run is still owed.

A partial run with an honest log is fine. A partial run masquerading as a full run is the failure mode the verification log is designed to catch.

The verification log

Every PR in the cycle gets a row in verification-log.md. A row that lacks evidence is treated as failing the tier-close checklist.

How AEP composes with other workflows

+ Panel of Experts

Architecture decisions during Phase 1 W3 freeze through two panel-review passes. The challenge phase produces the dissent the ADR records.

+ Handoff protocol

When ownership shifts between agents or sessions, the handoff protocol carries the AEP tracker state intact (last verification, next finding, open blockers).

+ TDD skill

Phase 2 PRs that touch behavior commit a failing test first (the "red" reference), then the fix, then the verification log entry that names both commits.

+ Session retrospective miner

An optional pre-Phase-1 step: mine the prior session window for narrative-quality scores below 50 and rewrite the phase specs before locking scope.

What to copy if you adopt this

If you want to run AEP outside EVOKORE, the smallest viable adoption set is:

  • phase-planning.md — a template for cycle metadata, phase goals, and dependency notes
  • test-matrix.md — the baseline + conditional + agent-specific matrix above
  • verification-log.md — the row-per-PR evidence table
  • templates.md — finding-ID format, branch/commit conventions, rubrics
  • A clear tier-close checklist that refuses to advance without evidence

The evaluation harness is optional but pays for itself by the second cycle.
