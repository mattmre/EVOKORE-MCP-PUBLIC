Workflows

Four representative flows

Skills compose. Two of these flowcharts show how code skills chain through a panel review into a PR. The other two show that the same composition pattern works for non-code work — turning rough ideas into PRDs, or running research reviews end-to-end.

Code — refactor a module Code — fix a bug Non-code — plan a project Non-code — research review

1. Code workflow — Refactor a module

An engineer wants to widen a heavily-imported module without breaking callers. The composition: repo-ingestor → panel-of-experts (architecture) → improve-codebase-architecture → pr-manager.

Each step writes a persisted artifact before handing off. The dotted box is a conditional panel — invoked only when the architecture review surfaces three or more viable interface shapes.

What happens at each step

  1. repo-ingestor walks the repo and produces a high-signal shape summary of the target context. The summary names the entry points, the most imported modules, and the obvious coupling hot-spots.
  2. panel-of-experts (architecture) reads the shape summary plus the candidate module. Four experts run SOLO reviews, then three CHALLENGE rounds at depth 3, then converge into a prioritized finding list. The output is a persisted markdown in docs/panel-reviews/.
  3. improve-codebase-architecture reads the panel finding list and the affected ADR. It applies the recommendations as a series of small, named commits. If the panel flagged 3+ viable interface shapes, this step pauses and triggers the design-an-interface panel before continuing.
  4. pr-manager packages the branch into a PR using the project template. The PR body links to the panel review SHA, the ADR, and the verification log. Reviewers inherit the context for free.

2. Code workflow — Fix a bug

A regression appeared in the rate limiter. The composition: triage-bug → tdd → pr-manager.

The TDD step is the heart of the flow. The verification log captures the red commit SHA before the green commit is allowed to merge.

What happens at each step

  1. triage-bug reads the session evidence log (and the repo audit, if available) to determine when the regression first appeared. It produces a markdown report at docs/bugs/<slug>.md with the suspected commit range, the failing scenario, and a proposed test seed.
  2. tdd writes the failing test first. The commit message captures the test failure. Only then is the fix attempted; the next commit names the red SHA explicitly. After green, an optional refactor pass cleans up without changing behavior.
  3. pr-manager opens the PR with both the red and green commit SHAs in the description, plus a link to the docs/bugs/ artifact. Reviewers can verify the regression is genuinely covered.

3. Non-code workflow — Plan a project

A team has been iterating informally on a feature direction for a few weeks. They want to lock the scope into a PRD and decompose it into actionable issues. The composition: to-prd → panel-of-experts (planning) → to-issues.

Each step writes a persisted artifact. The team can pause between steps without losing context — the PRD, the panel review, and the issue list are all on disk.

What happens at each step

  1. to-prd reads the recent session evidence — every recorded experiment, decision, and finding from the iteration weeks. It drafts a PRD with sections for problem, audience, scope, non-goals, success metrics, and risks. The skill auto-panels the draft before declaring it done; an unreviewed PRD never reaches "ready" state.
  2. panel-of-experts (Product Requirements panel) reads the PRD draft. Four experts run SOLO, with the product, engineering, growth, and devil's-advocate lenses. The persisted markdown lists the questions stakeholders are likely to ask and the gaps the PRD has not yet answered.
  3. to-issues reads the reviewed PRD and decomposes it into vertical-slice issues. Each issue carries acceptance criteria, an estimated effort tag, and an AFK / HITL flag (AFK = an agent fleet can take this without supervision; HITL = needs a human in the loop). The output is a set of GitHub issues, or a draftable issue list if the user prefers to review before filing.

4. Non-code workflow — Run a research review

A research lead has produced a 12-page market analysis. The team wants a structured review and a stakeholder briefing. The composition: content-research-writer → panel-of-experts (research) → meeting-insights-analyzer.

A research review uses the same three-node skeleton as the code workflows — produce, review, extract — with a panel review as the quality gate.

What happens at each step

  1. content-research-writer takes the raw research material and produces a long-form analysis. The skill separates research, synthesis, and writing into distinct phases so the synthesis is auditable independently of the prose. Output: a markdown report under docs/research/.
  2. panel-of-experts (research panel) reads the analysis. Four experts run SOLO: a methodology specialist, a domain specialist, a strategy specialist, and the devil's advocate. For an M&A-grade review, run at depth 5 so the concession ledger captures every shifted position.
  3. meeting-insights-analyzer reads the persisted panel review markdown and extracts the structured outputs that stakeholders actually need: the decisions made, the owners assigned, the deadlines, and the questions that remain unresolved. Output: a briefing file under docs/briefings/.

What all four flows have in common

Evidence-driven, not aspirational

Every flow starts by reading something on disk — repo files, session evidence, raw research. Nothing is invented from prompt context alone.

Persisted artifacts between steps

Each node writes a file before the next node reads it. The team can pause, audit, or hand off at any boundary without losing context.

Panel review as the quality gate

Three of the four flows include a panel review in the middle. The fourth (bug fix) substitutes TDD for the same role — a structured rigor that catches what a single perspective misses.

The shape generalizes

Produce → Review → Decompose, with a persisted artifact at each boundary. Use the same skeleton for any new workflow you want to add.
