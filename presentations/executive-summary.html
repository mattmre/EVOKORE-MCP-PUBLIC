Executive summary

One MCP endpoint. Many tools. Strong guardrails.

EVOKORE-MCP collapses multiple AI tool servers into a single, controlled, observable endpoint. It lets a small team get the productivity of dozens of integrations without losing the safety, repeatability, and operator visibility that an enterprise needs.

What it is, in plain English

Modern AI assistants like Claude, Cursor, and Gemini use a protocol called MCP (Model Context Protocol) to call tools — read files, search code, post to Slack, run a build. Each tool provider runs its own MCP server. Teams quickly end up with five, ten, or fifteen of them, each with its own configuration, permissions, and failure modes.

EVOKORE-MCP sits in front of those servers and presents one endpoint to the AI client. It does three things very deliberately:

  • Combines many tool servers into a single namespace, with safe, prefixed tool names that never collide.
  • Adds a human-in-the-loop approval step for any tool flagged as sensitive — no silent destructive actions.
  • Records every tool call, evidence artifact, and session purpose so the work is auditable and resumable.

Why it exists

Three problems show up repeatedly when an organization adopts AI agents:

Sprawl

Too many endpoints

Each new integration ships its own MCP server. The AI client juggles connections, retries, and config files. EVOKORE collapses them to one.

Cost

Too much tool context

Listing 300 tools eats the AI's context window. EVOKORE's dynamic discovery mode shows the tools the session actually needs.

Risk

Too little control

Letting an agent rename files, delete branches, or post to production is a governance gap. EVOKORE adds approval tokens, rate limits, and role-based access.

Who it is for

EVOKORE-MCP is built for the operator and the technical lead, not the casual user. Common profiles:

Engineering team

Long-running repo work

Teams that want AI to participate in real refactors, bug triage, and review — with the same audit trail a human PR gets.

Solo operator

Multi-tool power user

Anyone running Claude, Cursor, or Codex with more than three integrations who is tired of config drift and silent breakage.

Platform team

Tooling consolidator

Platform engineers who need a place to enforce permission, rate-limit, and approval policy across every AI tool the company uses.

Top five features

  1. One endpoint, many tools. Combine GitHub, filesystem, voice, database, and other MCP servers behind a single namespace with collision-safe prefixed names.
  2. Dynamic tool discovery. Choose between a full tool listing for compatibility, or a session-scoped activation flow that keeps the AI's context lean.
  3. Human-in-the-loop approval. Any tool tagged as sensitive returns an approval token instead of running. The AI asks the human, then retries with the token. No silent destructive actions.
  4. Role-based access control. Three roles — admin, developer, readonly — backed by a flat permissions file. Backwards-compatible with simple setups.
  5. Session continuity. A session manifest, replay log, evidence log, and task list under ~/.evokore/ mean a session can be paused, resumed, audited, and explained without losing context.

Top use cases

Engineering

Refactor a module with a panel review

Ingest the repo, send the candidate module to the architecture panel, apply the recommendations, and ship through a PR-manager workflow — all from one chat.

Engineering

Triage a production bug end to end

Triage the bug from session evidence, write a failing test, fix the code, and produce a PR with the verification log already attached.

Planning

Turn a rough idea into shippable issues

Synthesize a PRD from prior session evidence, run the planning panel against it, and decompose into vertical-slice issues with explicit accept criteria.

Research

Run a research review with diverse personas

Convene a four-expert panel against a research artifact, capture challenges and concessions, and persist the result as a dated review for the team archive.

Operations

Approve sensitive actions safely

Configure the filesystem and GitHub writers as approval-required, then let the AI propose changes. Every destructive call is gated and logged.

Knowledge

Find the right workflow in 5 seconds

Ask "I need to wrap this session cleanly" and EVOKORE resolves the matching skill, explains why it matched, and injects the steps directly into the response.

What new users trip on

  1. Node 18+ is required. Older versions silently misbehave because the MCP SDK depends on modern Node features. SEV 5
  2. Child servers boot in the background. The AI handshake completes before the proxied tools are ready. First-second calls may not find them. SEV 4
  3. Default discovery is now dynamic. In v3.1+ the legacy full tool listing is opt-in. Set EVOKORE_TOOL_DISCOVERY_MODE=legacy if a client expects every tool visible up front. SEV 4
  4. Approval tokens are one-shot and arg-bound. Retry must use the exact same arguments. Changing a single character invalidates the token. SEV 3
  5. Tests use vitest now, not the old node test-*.js chains. Run npx vitest run instead of the legacy entry points. SEV 3
  6. Voice and dashboard run on separate ports. The sidecar is on :8888, the dashboard on :8899. Conflict with another local service and they will fail to bind. SEV 3
  7. Optional child servers need their own keys. ElevenLabs, Supabase, and similar integrations are disabled until their API keys exist in .env. SEV 2

What's not in scope

EVOKORE-MCP is intentionally narrow. It does not try to:

  • Replace the AI client. It runs underneath whatever client a team already uses.
  • Train or fine-tune models.
  • Provide a hosted SaaS — it's a local process for now.
  • Cover non-MCP integrations directly. Those have to be wrapped as an MCP server first.
