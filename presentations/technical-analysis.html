Technical analysis

The runtime, in detail

EVOKORE-MCP is a TypeScript stdio MCP router on Node 18+. It merges 37 native tools with a configurable set of proxied child MCP servers, projects that combined surface in either legacy or dynamic mode, and applies RBAC, rate limiting, and async proxy boot on top.

System architecture

Everything an AI client sees passes through a single stdio transport. Behind that transport, native tools and proxied child servers cohabit one unified catalog.

A single stdio transport carries every tool call. Native managers and the proxy manager hand back into one merged catalog projected to the client.

Transport and discovery modes

Transports

EVOKORE itself speaks stdio to the AI client. Child MCP servers can be reached via two transports:

  • stdio — the common case. EVOKORE spawns the child process and pipes JSON-RPC over stdin/stdout.
  • HTTP (StreamableHTTPClientTransport) — for hosted child servers. Set "transport": "http" and "url" in mcp.config.json.

Discovery modes

The same set of tools can be projected to the AI client two ways:

ModeBehaviorWhen to use
legacytools/list returns every native + proxied toolOlder clients that don't poll for tools/list_changed, or sessions where compatibility beats context economy.
dynamictools/list returns native tools + only the proxied tools the current session has activated via discover_toolsFocused sessions, clients sensitive to tool-list size, contexts where 200+ tools would otherwise crowd the window. Hidden tools remain callable by exact prefixed name.

Every native tool carries MCP annotations (readOnlyHint, destructiveHint, idempotentHint, openWorldHint) plus a human-readable title, so MCP-aware clients can render appropriate confirmation UI.

Tool call lifecycle

Native tools dispatch in-process. Proxied tools pass through the security manager; sensitive ones return an approval token instead of running.

Async proxy boot

EVOKORE completes the MCP handshake before any child server has finished booting. Background boot avoids holding the client hostage on the slowest dependency:

  1. Load .env and initialize the MCP server capabilities.
  2. Index skills from SKILLS/ (optionally watched).
  3. Connect stdio transport and start serving requests immediately.
  4. In the background: load mcp.config.json, resolve ${VAR} placeholders, spawn each child, fetch its tool list, prefix and register, rebuild the catalog.
  5. Emit "Proxy bootstrap complete" or "Background proxy bootstrap failed" on stderr.

Native tools are usable instantly; proxied tools appear as their child server finishes. Configure the boot timeout with EVOKORE_CHILD_SERVER_BOOT_TIMEOUT_MS (default 30 seconds).

RBAC and rate limiting

Roles

RoleDefault permissionEffective behavior
adminallowFull access to all proxied tools.
developerrequire_approvalReads allowed, writes gated, destructive denied unless overridden.
readonlydenyOnly explicitly overridden reads are allowed.

Set EVOKORE_ROLE to opt in. When unset, EVOKORE falls back to the flat rules: block — fully backwards-compatible.

Rate limiting

A token-bucket rate limiter runs per child server (and optionally per tool). Configure maxTokens, refillRate, and refillIntervalMs under rateLimit in mcp.config.json. It is independent of the error-triggered cooldown that ProxyManager applies to flapping upstream tools.

Resources and prompts

EVOKORE exposes MCP resources and prompts on top of tools:

  • resources/list returns skills as skill://{category}/{name} URIs plus server-level resources (evokore://server/status, evokore://server/config, evokore://skills/categories).
  • prompts/list returns three prompts: resolve-workflow, skill-help, server-overview.
  • resources/read and prompts/get return the underlying content with consistent shape so clients can render them uniformly.

Session continuity

State that survives a restart lives in ~/.evokore/, not in the MCP process. The canonical manifest is at ~/.evokore/sessions/{sessionId}.json and is the single source of truth for purpose, lifecycle, and artifact pointers.

What new users trip on

  1. Schema-deferred mode is opt-in and not safe for most clients. Leave it off unless you control the client and have tested it. SEV 5
  2. tools/list_changed is best-effort, not guaranteed. Some clients ignore it and need a manual tools/list refresh after discover_tools. SEV 4
  3. Windows command resolution differs. Only npx is remapped to npx.cmd; uv and uvx must already resolve on PATH. SEV 4
  4. Unresolved ${VAR} placeholders fail fast for that child server only. The router stays up; missing-env errors do not surface unless you read stderr. SEV 4
  5. Cooldown is not the same as rate limiting. Cooldown reacts to repeated upstream errors and can mask real failures. Inspect with proxy_server_status. SEV 3
  6. Skill watcher is opt-in. Set EVOKORE_SKILL_WATCHER=true to auto-reindex; otherwise call refresh_skills manually. SEV 3
  7. HITL tokens are one-shot and exact-args-bound. Reissue if the retry changes any argument. SEV 3
